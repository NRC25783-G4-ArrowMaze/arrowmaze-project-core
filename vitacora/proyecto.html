Cómo se hizo el proyecto

Un puzzle game construido spec primero, código después

Arrow Maze es un puzzle de flechas sobre un tablero-grafo. Pero antes que una línea de código, existió una especificación: cada regla del juego se discutió, se decidió y se congeló en Gherkin antes de implementarse. Esta página cuenta la arquitectura, el stack, la metodología y el equipo detrás de ese proceso.

Forma de trabajo

Specification-Driven Development (SDD)

El equipo trabaja con una regla central: project-core es la fuente única de verdad. Las decisiones de diseño se toman de forma iterativa mediante Q&A estructurado con IA, se sintetizan en specs Gherkin, y solo entonces el cliente y el backend las implementan. Nada se implementa "sobre la marcha": todo cambio se traza de vuelta a una sesión de diseño.

Elicitación

Claude estructura preguntas para desambiguar cada decisión de diseño. Entregable: una matriz de decisiones (cerradas / abiertas). Ej.: ¿penalización lineal o exponencial por fallas consecutivas?

Consolidación

De las decisiones validadas nace un archivo .feature (Gherkin/BDD, en español): escenarios, background, tablas de ejemplos e invariantes explícitas.

Revisión & síntesis

El diseño se valida (casos borde, topología, dependencias) antes de que los repos de implementación empiecen. Calidad sobre velocidad.

◇ BDD en Gherkin, en español

Cada feature es una spec ejecutable escrita en Given-When-Then (Dado-Cuando-Entonces), con tablas y escenarios. La idea: los tests de dominio mapean 1:1 contra los escenarios Gherkin.

  • Dado un tablero 3×3 con flecha en C0 apuntando al puerto 1
  • Cuando se ejecuta un movimiento
  • Entonces la flecha avanza a C1 y su cabeza queda en el puerto 3

◇ Uso de IA trazable

Todo el rationale de diseño vive en .ai-usage/: fecha, modelo, duración, decisiones tomadas y alternativas rechazadas de cada una de las 28 sesiones. Modelos usados: Claude (Sonnet 4.5/4.6, Opus 4.7/4.8, Fable 5) y Gemini.

  • SDD exige trazabilidad decisión → diseño → spec → implementación
  • El registro completo es la Bitácora de IA de este sitio

Política de versionado y congelamiento. project-core documenta cada repo de implementación hasta su v1.0.0. Cuando un repo cierra esa versión, su documentación aquí queda congelada y sus versiones futuras evolucionan de forma independiente. Ambos repos ya están congelados en v1.0.0 — el backend desde el 2026-07-09 y el cliente desde el 2026-07-11. Este repo funciona desde entonces como bitácora histórica del cómo se llegó a la v1.0.0.

Diseño del sistema

Clean Architecture + Domain-Driven Design

La separación en capas es innegociable. El dominio no depende de nada; todo lo demás depende hacia adentro. Esto mantiene las reglas del juego puras, testeables en aislamiento y reproducibles.

Presentación
React — renderizado SVG, input, pantallas
Aplicación
Casos de uso orquestadores (PlayMoveUseCase…)
Adaptadores
Repositorios, adaptadores de UI y API
Dominio
Entidades puras — nunca depende de nada

▲ cada capa depende solo hacia abajo · el dominio no importa nada externo

◆ Decisiones arquitectónicas clave

  • GameSession como agregado raíz: gestiona movimientos, estado y score; no contiene Board ni Arrow[] directamente
  • PlayMoveUseCase envuelve a AdvanceArrowUseCase sin extenderlo (SRP)
  • Score como Value Object inmutable, calculado al final de la partida
  • Tablero como grafo: recorrido O(1) con aritmética modular; puertos indexados, no direcciones cardinales
  • Ticks deterministas: el tiempo se mide en ticks, no en segundos de reloj (tests reproducibles)

▦ Tablero & grafo

Celdas = nodos, puertos = bordes. Puertos indexados 0…P-1 (P siempre par). Puerto opuesto: (entrada + P/2) % P. La topología es agnóstica a la forma visual (cuadrado, hexágono…).

➜ Movimiento de flechas

Todas las flechas avanzan en el mismo tick (movimiento simultáneo). Solo la cabeza chequea colisión. Resultado blocked o success; el movimiento es atómico: éxito o fallo como unidad.

★ Puntuación (score)

max(0, BASE − ticks×DECAY − Σ penalizaciones + bonus flawless). Solo visible en estado WON; nula en LOST/IN_PROGRESS. Nunca negativa, sin techo. Flawless = cero fallas + victoria.

Tecnología

Stack elegido y los tres repositorios

El stack se eligió con una matriz de decisión ponderada (8 criterios × 6 tecnologías). React Web ganó con 8.73/10 frente a Flutter (7.58), por compilación rápida en hardware modesto, testing sin emulador, excelente soporte de TypeScript en ambos extremos y buen encaje con Clean Architecture.

⚛ Frontend / móvil

React 18 + TypeScript + Vite (compilación 1–2 s). Capacitor reutiliza la web app para empaquetar en Android/iOS. Renderizado del tablero en SVG (decisión NQ4).

⛃ Backend / datos

Express.js sobre Node — mismo lenguaje (TypeScript) en ambos lados. PostgreSQL en producción y SQLite para local/testing. El backend materializa patrones GoF (Factory, Singleton, Adapter, Strategy) y 2 aspectos AOP, con OpenAPI/Swagger en /api/docs.

✓ Testing & CI

Jest + React Testing Library (frontend), Jest + Supertest (backend). CI de build+tests en cada PR vía GitHub Actions. Los tests de dominio mapean 1:1 contra los escenarios Gherkin.

Historia

Hitos del proyecto

De la selección de tecnología al proyecto congelado en v1.0.0: 28 sesiones SDD entre mayo y julio de 2026. La historia completa, sesión por sesión, está en la Bitácora de IA.

2026-05-03

Selección de stack

Matriz de decisión ponderada (8 criterios × 6 tecnologías). React Web gana con 8.73/10 frente a Flutter (7.58). Se define Clean Architecture + DDD como patrón obligatorio.

Mayo 2026 — 8 sesiones

Fundación & arquitectura

Capas de dominio, agregado GameSession, tablero como grafo, invariantes de puertos y ticks deterministas.

Junio 2026 — 8 sesiones

Lógica del juego & specs del motor

Movimiento, detección de fin de partida y fórmula de puntuación (A1–A5). Punto de control del 15 de junio: project-core se declara fuente única de verdad.

Julio 2026 — 12 sesiones

Flujo, presentación y herramientas

Specs de C1 (máquina de estados), C3 (selección de niveles), G1–G3 (audio/i18n/timer) y H1 (FORGE). Renderizado resuelto a SVG.

2026-07-09

Backend cerrado en v1.0.0

Patrones GoF + AOP + OpenAPI/Swagger mergeados (PR #17) y liberados como v1.0.0. El backend queda congelado en este repo; el cliente cerraría dos días después.

2026-07-11

Cliente cerrado en v1.0.0 — proyecto congelado

El cliente libera su v1.0.0 (release #62): motor completo, persistencia local + sincronización, identidad, leaderboard, audio, i18n, temporizador, tema claro/oscuro, tutorial guiado y FORGE. Con ambos repos congelados, project-core queda como bitácora histórica del proyecto.
